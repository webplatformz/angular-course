Refactoring I: Components

Types of Components

The larger a project, the more components we build. Taking it component-by-component turns the application into a unmanageable sea of components.

Why is it a problem, if all components «look the same»?

  • mental model is overwhelming: keeping track of data flow and dependencies gets increasingly more difficult, more effort to onboard new team members, new features take longer, changes are riskier
  • debugging is harder: more places where responsible code could be
  • testing is more complex

Structure your components

Delegate different jobs to different components. There are multiple ways to differentiate components, the easiest one is:

  • smart components: contain (business) logic, handle user events, interactive
  • dumb components: also called container components, they only bootstrap other components and build the connection between them, layouting
